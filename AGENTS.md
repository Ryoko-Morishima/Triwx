> 最初に PROJECT.md（特に §2 設計原則と §6 教訓）を読んでから本書に従うこと。

## 0. 最初に読む順番

1. `PROJECT.md` — 目的・設計原則・教訓（最重要）
2. 本書
3. `HANDOFF.md` — 実装の現在地とファイル地図

## 1. 役割分担

**基本原則: 人間（プロデューサー）は細かい仕様を決めない。**

### 人間が決めること
- 体験の良し悪しの最終判断（「なんか違う」の言語化を含む。人間の違和感は仕様より上位の入力）
- 耳での主観評価（カード適合、音の質感、ナレーションのトーン、ジングルの好み）
- 設計原則の変更・フェーズ移行・方向転換
- コスト増を伴う判断（LLMモデル変更、外部API追加、有料プラン）

### AIが決めてよいこと（事前確認不要）
- 上記以外のすべての実装詳細: 関数設計、データ構造、プロンプト文言、閾値の初期案、リファクタリング

### 人間に確認すべきこと
- 設計原則（PROJECT.md §2）と衝突しうる変更
- 「触ると危ない領域」（§3）への変更
- 音・声・見た目の趣味に関わる変更（実装してから聴いて/見てもらうのは可。無断で確定しない）
- 外部依存の追加、Spotify/OpenAIの利用形態の変更

## 2. 不変条件（壊したら差し戻し）

1. **実再生**: モック再生・再生のふり・プレビュー代替への退行は禁止
2. **無音にしない（最優先）**: fill APIは多層フォールバックで必ず1曲返す。条件が厳しくて満たせないときは、条件を段階的に緩めてでも鳴らす（緩めたことはfallbackReasonでログに残す）。この優先順位は仕様であり、フォールバック発動はバグではない
3. **アーティスト不一致の棄却**: 誤解決防止の生命線。どの層でも緩めない
4. **評価=カード適合**: ◎✕を好み学習・レコメンドに使う機能を作らない
5. **除外の減衰**: 永久ブラックリストを作らない（7日減衰・セッション内全曲、の現行設計を維持）
6. **秘密情報**: `.env.local`・APIキー・トークン類を**絶対にコミットしない**。コードへの直書き・ログ出力も禁止。`.gitignore` を弱めない
7. **語彙の単一ソース**: カード/スライダーの意味は `src/pipeline/definitions.ts` 以外に書かない
8. 機能を伴わない意匠を足さない（針・ランプ・音は本物を指す）

## 3. 触ると危ない領域

- **`src/app/api/auth/`・`src/server/spotifyAuth.ts`**: PKCE認証。本番・ローカル両方で安定稼働中。明確な必要がない限り触らない。触る場合はローカルと本番URLの両方でログイン確認必須
- **`src/engine/useRadioEngine.ts` の状態遷移（transitionToNext/ポーラー）**: 曲末検出・飢餓復帰・二重遷移ガードが絡む。変更時は「1時間連続再生で止まらない」ことを確認
- **Web Playback SDKまわりの注意**:
  - SDKの初期化・AudioContextの解錠は**ユーザー操作（開始ボタン）起点**でしか動かない。自動開始を作らない
  - 曲末はSDKのイベントに頼らず残り時間ポーリングで検出している（イベントは信頼できない）。この方式を維持
  - `player.getCurrentState()` はnullを返しうる。防御を外さない
  - Premiumでないアカウントは `account_error` で落ちる（仕様）
- **`src/logs/store.ts` のNDJSON/PG二層**: スキーマ変更時は `schema.ts`・本書・HANDOFF.md を同時更新。既存ログを壊すマイグレーションは人間に確認

## 4. 変更後に必ず確認すること

区分に応じて:

**すべての変更**: `npm run typecheck` / `npm run build` が通る。`git status` に `.env.local` や `data/sessions/*.ndjson` が現れない

**選曲結果に影響する変更**（プロンプト・パイプライン・ゲート・definitions）:
- ローカルで実再生し、10曲以上流す
- ログで確認: source（seedが常態化していないか）、judgeRejected（全滅していないか）、重複、fallbackReason
- 可能ならユーザーに耳評価を依頼（カード適合は最終的に人間の耳でしか判定できない）

**再生・演出に影響する変更**: 開始→曲→曲間→ナレーション→次曲の1サイクル+スキップ+一時停止/再開を実機で確認。読み上げ3モードすべてで曲間が壊れないこと

**デプロイに影響する変更**: 環境変数の増減はREADMEとHANDOFFに反映。Vercelでは再デプロイまで環境変数が反映されないことを人間への指示に含める

## 5. 作業記録（引き継ぎの生命線）

- タスクは `tasks/*.md` を単位とし、着手時にステータスを `in-progress` に、完了時に `done` に更新
- **中断・キャップ時は必ず「中断時の引き継ぎ欄」に現在地と次の一手を書く**。書けないほど急な中断に備え、大きな変更は小さく刻んでコミットする
- コミットメッセージにタスクID（例: `T4:`）を付ける
- 選曲挙動を変えたら PROJECT.md §6（教訓）に学びを追記する

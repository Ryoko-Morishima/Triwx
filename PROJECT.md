## 1. このアプリの目的

TRIWXは、AIが次の一曲を選びつづける**個人ラジオ局**である。
番組の開始も終了もなく、途中でつけて途中で消してよい。音楽が流れつづけ、
曲間に短い曲紹介（事実ベースの解説）が入り、ときどき時報とジングルが流れる。

**第一目的は「聴いたことのない曲との遭遇」。**
好きな曲を聴くだけならSpotify本体で足りる。TRIWXの存在理由は発見にある。

## 2. 何を重視するアプリか（設計原則・ユーザーと合意済み）

1. **発見**: 第一目的は未知の曲との遭遇。既知への最適化はしない
2. **カード適合の厳密さ**: ユーザーが指定した空気（カード・スライダー）は厳密に守る
3. **条件内で最大限散らす**: 散らばり（アーティスト重複率・年代幅・言語分布）はログから客観的に測る。「知ってる/知らない」は個人差なので指標にしない
4. **無音にしない**: どんな失敗時も音楽が鳴りつづけること。条件遵守より放送継続が優先（多層フォールバックは仕様であってバグではない）
5. **好み最適化アプリではない**: ◎○✕は「好き嫌い」ではなく**「いま挿さっているカードにこの曲は合っていたか」の判定**。好みの学習・パーソナライズには使わない（好みに最適化すると既知領域に収束し、第一目的を裏切る）
6. **除外は減衰する**: セッション内は全曲除外、セッション跨ぎは直近7日のみ。**永久ブラックリストは作らない**（「散らすための除外」が「定番が二度とかからない呪い」にならないため）
7. **針・ランプ・音は嘘をつかない**: 機能を伴わない意匠を置かない（例: FM周波数風の飾りダイアルは却下済み。文字盤は実際の経過時間計。時報のチャイムも正時を主張しない鐘）

## 3. 現在できていること（2026-07-05）

- Spotify Web Playback SDK によるフル尺の連続実再生（要Premium）
- AI選曲パイプライン: 方針生成 → 候補生成 → 審査パス（カード適合の検品）→ Spotify解決（アーティスト不一致棄却・原曲優先）→ 年代/言語ゲート
- 多層フォールバック（拡大バッチ→年代緊急緩和→リプレイ→種曲）で放送を止めない
- 曲間ナレーション（事実ベース解説。読み上げ: オフ/ブラウザ/OpenAI TTS高音質の3段）
- カード（雰囲気20枚・ことば地域7枚。時間帯/天気は排他）とフェーダー3本（年代・温度・人気度）。再生中変更→次の選曲から反映、方針は事前ウォームアップ+旧バッファ整理
- 時報（毎時55分〜の曲間で「◯時まであとおよそ◯分」+柔らかい鐘）とジングル（合成サウンドロゴ+局名コール、開局時にも再生。声は読み上げ設定から独立・増幅済み）
- 評価（◎○✕=カード適合）とメモ、1曲1行のログ（ローカル: NDJSON / 本番: Neon Postgres 自動切替、冪等、userId付き）
- 言語ドリフト検知・是正（日本語吸着のハードゲート）、7日減衰の重複除外、アーティスト10曲クールダウン
- Vercel本番デプロイ済み（https://triwx.vercel.app）、/about ページ、ファビコン

## 4. まだできていないこと

- **散らし具合レポート**: 重複率・年代幅・言語分布のログからの自動集計（設計原則3の測定装置。tasks/T4）
- 日本語以外の言語圏への吸着検知（スクリプト判定はJPのみ。西・韓等はプロンプト対策のみ）
- 「深掘り」の客観検証（Spotifyの人気度メトリクスはAPI制限で利用不可。現状プロンプト+審査が上限）
- 発売年データの品質（コンピ盤・再発盤の年。リイシュー規則で緩和済みだが完全ではない）
- 複数ユーザーの本格運用設計（Spotify規約で最大5人。ログはuserId付きだがUI上の区別なし）
- ナレーション音声のさらなる品質（声の個性、抑揚）
- タブを閉じても続く再生（現状ラジオはブラウザタブの中に住んでいる。仕様として許容中）

## 5. 今後の方向性

短期は安定化と観測（本番ログでフォールバック率・散らばりを見る）、中期は演出の磨き込みと散らし測定の自動化、長期は少人数共有の体験設計と選曲検証の高度化。詳細は `ROADMAP.md`、作業単位は `tasks/` を正とする。

## 6. 開発で得た教訓（新規参加者は必読）

実測ログ（累計150曲超の分析）とユーザー評価から確定した知見。**同じ穴に落ちないこと。**

- **プロンプトの言い聞かせは長期戦で負ける**。長期挙動（重複・吸着・年代）はコードのゲートで守る。特に、プロンプトに「最近流れた曲リスト」を入れると強力な実例集として働き、方針より直近の傾向を模倣させる（言語吸着とカード減衰の共通原因だった。重複排除はコードで行い、履歴はプロンプトに入れない）
- **検証を厳しくしすぎると非常口が開く**。審査に人気度まで判定させた結果「知ってる曲=定番でNG、知らない曲=不明でNG」のデッドロックが起き、seed発動15%・「踊れる」放送にFrank Sinatraが流れた。審査はカード+極値温度のみ、知らない曲は通す（実在確認はresolveの仕事）
- **LLMは語面で選ぶ**。「ドライブ」カードで曲名にdrive/road/rideを含む曲へ吸着する。「質感で選ぶ、語面一致は禁止」の明示が必要
- **Spotify検索の上位を無条件に取らない**。remix/live/cover/Take◯が原曲より上に来る（原曲優先スコアリング導入済み）
- **中立=「自由」と書くとLLMは最有名曲に沈む**（定番重力。中立帯の文言にも幅を持たせる指示が要る）
- **同じ言語圏の連続は「接続」の最短経路**。接続とは質感・温度・リズム・時代のことだと定義し、言語・国籍を明示的に除外する
- **環境変数は再デプロイまで反映されない**（Vercel）。APIキーは改行・空白混入でHTTPヘッダごと壊れる（2026-07に本番で実際に発生。コード側でtrim防御済み）

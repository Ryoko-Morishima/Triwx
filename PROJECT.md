# PROJECT.md — TRIWX 現状認識（正）

> このリポジトリの現状認識の正。コード変更時は本書も更新すること。最終更新: 2026-07-04

## 設計原則（最上位・ユーザーと合意済み 2026-07）
1. **第一目的は「聴いたことのない曲との遭遇」**。好きな曲を聴くだけならSpotify本体で足りる。TRIWXの存在理由は発見にある
2. **評価（◎○✕）は「カード適合」の判定であり、個人の好悪ではない**。好みの学習・パーソナライズには使わない（好みに最適化すると既知領域に収束し、第一目的を裏切るため）
3. **カード条件は厳密に守り、その条件の中では最大限散らす**。散らばり（アーティスト重複率・年代幅・言語分布）はログから客観的に測る。「知ってる/知らない」は個人差なので指標にしない
4. **除外は減衰する**。セッション内は全曲除外、セッション跨ぎは直近7日のみ。永久ブラックリストは作らない（「散らすための除外」が「定番が二度とかからない呪い」になるのを防ぐ）
5. 針・ランプ・音が指すものは常に本物であること（機能を伴わない意匠を置かない）

## ゴール
「自分専用の連続ラジオ」。番組の開始・終了がなく、途中からつけてよく、音楽が流れ続け、曲間に短いつなぎコメントが入る。ただのプレイリストではなく「誰かが選んでつないでいる」気配があること。

## 設計の中心思想
- **ストリーム型**: 大量選曲→絞り込みではなく、少量（1曲ずつ）を高頻度で継ぎ足す
- **小さいバッファ（先読み2曲）が体験そのもの**: 設定変更が1〜2曲で効き始める応答性と「流れが少しずつ変わる」感じを両立させる
- **生成は冒険・検証で守る**: 候補生成は自由に、Spotify解決でアーティスト不一致を棄却
- **語彙の単一ソース**: カード/スライダーの意味は `src/pipeline/definitions.ts` のみに書く。プロンプトも（将来の評価も）ここから導出する
- **地域の脱・英語圏バイアス**: 「ことば・地域」カード（最大2枚）は雰囲気カードと別格の【最優先・厳守】条件としてプロンプトに渡す。未指定時も候補生成に「英語圏だけに偏らない」指示を常時入れている

## 構成
- `src/engine/` — クライアントのラジオエンジン（状態機械: playing/narrating/starved、残り時間監視、補充トリガー、読み上げ3モード: オフ/ブラウザTTS/OpenAI TTS）
- `src/pipeline/` — サーバの選曲パイプライン（definitions=語彙 / core=方針・候補・解決・ナレーション）
- `src/logs/` — 1曲=1行のログ（冪等追記+パッチ）。DATABASE_URLがあればPostgres（Neon）、なければローカルNDJSONの二層。人間評価・userIdもここに載る
- `src/app/api/` — auth(PKCE) / token / queue/fill / log

## 処理フロー
起動→fill(1曲)→ナレーション読み上げ→再生。再生中は700ms間隔で残り時間を監視し、残り45秒でバッファ<2なら/api/queue/fill。fillは 方針(versionごとにLLM1回・キャッシュ)→候補4件(LLM)→Spotify解決(不一致棄却・最大2バッチ)→ナレーション(LLM)→ログ下書き→QueueItem返却。曲末（残り1秒）で遷移: 前曲をplayed記録→次のナレーションをTTS→次曲再生。キュー0で曲が終わればstarved表示で補充を待つ。

## 設定変更の反映
カード/スライダー変更→500msデバウンスでstateVersion++。バッファ済みの曲は触らず、次のfillから新方針。ナレーションにはstateChangedフラグが渡り転調の一言が入る。スキップはskippedとしてログに残る（暗黙の負評価）。

## ナレーションの方針（ユーザー評価で確定済み）
- 次の曲の事実解説のみ。前曲への言及・流れの説明・カード適合の説明・呼びかけは禁止
- 使ってよい事実: 与えた年/アルバム名 + モデルが確実に知る一般事実（出身、ジャンル、活動時期、音楽的特徴）。チャート・売上・受賞・逸話は禁止
- 読み上げオフ時は間をあけず即次曲（テキストは表示継続）

## UI意匠（確定）
「深夜のトランジスタ」: 木目筐体 + クリーム文字盤。文字盤の目盛りはその曲の時間軸（15秒刻み・分ごとに数字。曲が変わるたびに引き直される）で、赤い針は経過時間計として正直に動く（※当初のFM周波数目盛り案は「機能を伴わない意匠」としてユーザー評価で却下済み。針が指すものは常に本物であること）。スライダーはフェーダーで、現在の帯ラベル（「指定なし」等）を生表示。カードは紙メタファー: 下段カードボックス（4分類タブ）から選ぶと本体上スロットに索引カードとして挿さる。ことば・地域カードは青系の紙。ナレーション中はアンバーの「放送中」ランプ点灯。

## 言語ドリフト・カード減衰対策（構造化済み）
症状: 日本の曲が1曲入ると日本語圏が続く／長いセッションでカードが効かなくなる（ユーザー報告で確認。プロンプトの言い聞かせのみの初期対策は長期戦で敗北）。
根本原因: 候補生成プロンプトに入れていた「最近流れた曲20件」が、セッションが伸びるほど強力な実例集（few-shot）として働き、方針・カード・是正指示を数で圧倒していた。言語吸着とカード減衰は同一の病。
構造対策（現行）:
1. **履歴の壁を撤去**: 重複排除はプロンプトでなくサーバ側コードで実施（historyとの完全一致・直近アーティスト連発をfillで棄却）。プロンプトに履歴リストを入れない
2. **選曲条件の毎回注入**: describeState()の生テキスト（カード・スライダー・地域厳守）を全候補生成に【厳守】として渡し、「条件は直近の傾向より常に優先」と明示
3. **言語ハードゲート**: ドリフト検知中（直近3曲中2曲がJPスクリプト・地域カード未指定）は、日本語候補をコードで棄却。リプレイ層・種曲層にも同じ回避を適用
4. **アンカー切断**: ドリフト是正中は「直前の曲」をプロンプトから外し、方針の中心から選び直させる
発動はログの driftBreak フィールドで追跡可能。日本語以外の言語圏への吸着はスクリプト判定では検知不可（プロンプト対策のみ有効）。将来候補: Spotifyメタデータでの言語・市場検証。

## カード質感の検証（審査パス）
症状: 「踊れる+高揚」でR.E.M.等、条件に薄く合うだけの曲が混入（ユーザー報告）。原因は質感に検証工程がないこと（年代・言語はゲートがあるがカードは生成任せだった）。Spotifyのdanceability等の音響API（audio features）は2024年11月に廃止済みのため機械検証は不可。
対策: 候補生成のあと、カード選択時のみ「審査パス」（judgeCandidates, temperature 0.1）を通す。判定専任のLLM呼び出しが各候補を条件全件と突き合わせ、1つでも外れる/判断がつかない/曲を知らない候補を解決前に棄却。審査失敗時は無審査で通す（放送優先）。棄却数はログの judgeRejected に記録。あわせて「踊れる」カードに不可例（テンポが速いだけのロック等）を明記。コスト: fillあたりgpt-4o-mini +1呼び出し。

## セッションログ分析による一括改善（2026-07 実測40曲の分析に基づく）
発見された連鎖: 審査が厳格すぎ（人気度まで判定+知らない曲=不可）→候補全滅→seed非常口が15%発動→条件無視の定番曲混入。これが「深掘りが浅い」「クール指定が効かない」体感の主因。加えて重複排除キーの表記ズレ（remix接尾辞等）で同一曲がすり抜け。
対策:
1. 審査対象をカード+極値の温度感のみに限定（describeJudgeConditions。人気度・年代は含めない）。「知らない曲は通す」に変更（実在確認はresolveの仕事）
2. 重複排除をnormalizeNameの正規化キーに変更し、解決後の正式表記でも再チェック
3. アーティスト・クールダウンを直近2曲→10曲に拡大
4. 解決の原曲優先スコアリング（remix/live/cover/edit等を強く回避、リマスターは軽く回避）
5. リイシュー年代規則: リマスター等は「新しめ」指定時は不可、「古めのみ」指定時は可
6. seed層にも年代ゲート・重複排除・クールダウンを適用（全滅時のみ2周目で無条件、fallbackReason=seed_unconditional）
7. 温度感の再定義（ユーザーの言葉: クール=感情抑制、ホット=感情+肉体感。両端に不可例を明記）
8. 深掘りの強化（「インディーの定番も定番」を明記）※Spotifyの人気度メトリクスは2026年2月のAPI制限で利用不可のためプロンプト+審査での対応
9. 意外性スライダーを廃止（効果が指示文1行のみで体感不能なため。「機能を伴わないレバーを置かない」原則）
10. ジングル不再生バグ修正: AudioContextを再生毎に生成→suspendedで無音になっていた。開始操作時にunlockAudioで解錠し共有コンテキストを使い回す。ジングルをFM風サウンドロゴに再設計（パッド+上昇モチーフ+エコー → 局名コール「TRIWX STATION」 → 締めスパークル）

## 一括修正（2026-07 実測127曲の分析とユーザー評価に基づく）
- **重複の全セッション化+7日減衰**: 旧実装は直近20曲窓のみで長回しで同一曲が回帰（WALK THE MOON 21曲差, Stooges 89曲差で実証）。セッション内全曲+全セッション横断の直近7日を除外し、7日経過でプール復帰
- **語面連想の禁止**: ドライブ→曲名にdrive/road/ride等、カードの単語がタイトルに入る曲への吸着を実測。プロンプトで明示禁止
- **中立人気度の重力対策**: 指定なしでも超定番に沈む傾向への文言調整
- **バージョンマーカー拡充**: Take◯/outtake/alternate/anthology等（Beatles "Take 9" 混入の対策）
- **時報の作法**: 発火窓55分〜に短縮、チャイムは時刻を主張しない柔らかい鐘に変更、文言は発話時点の残り分数を正確に告げる（「23時まで、あとおよそ4分です」）
- **サウンドロゴの声を読み上げ設定から独立**: 局名コールは番組演出であり、開始時に音声を先読みキャッシュして即再生。開局時にもサウンドロゴを鳴らす（初回fillの待ち時間を隠す演出を兼ねる）
- **方針の事前ウォームアップ**: /api/queue/warm（共有policyCache）。カード変更確定時に裏で方針生成を先行、あわせて旧条件のバッファを1曲残して破棄・先読み（変更が効くまでの体感遅延の短縮）
- **時間帯カードの排他**: 朝/昼下がり/夕暮れ/夜更けは択一（選ぶと入れ替え）。雨/晴れた休日も同様
- **ランプの意味論**: 消灯=停波 / 点灯=音楽 / 発光=マイク（ナレーション・時報・ジングル・選局中）。読み上げオフでも曲の切り替わりに2.5秒発光して曲間を可視化（音は止めない）
- **一時停止の誤スキップ防御**: 「停止かつ位置0」の終了判定は再生実績30秒超のときのみ発火
- **評価UIの自己説明化**: ラベルを「カードに合ってる?」に変更（設計原則2の反映）
- **/about ページ**: アプリの目的・カード/評価の意味・注意事項。本体フッターからリンク

## 公開運用の注意
- /api/tts /api/log /api/queue/fill は全てSpotify認証ガード済み（未認証の第三者はOpenAIキーを消費できない）
- 方針キャッシュはインスタンス内メモリ。serverlessでは失われることがあるが、再生成されるだけで実害なし
- Spotify Dev Mode: 許可5ユーザー・全員Premium・開発者もPremium必須（2026-02の規約変更）

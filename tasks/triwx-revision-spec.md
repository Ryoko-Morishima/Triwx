## 背景(要約)

- bad 15件の主因は3系統: (a) 時間カードの定義がアレンジベースで情景とずれる(midnight事故9件)、(b) 3枚使い時に浮いた1枚が黙って無視される(grit不足3件)、(c) パイプライン起因(replayのカード無照合、Spotify解決のバージョン事故)。
- 数値ゲート(judge)の却下回数と人間評価は無相関(bad率: 却下0回=25%、1-2回=21%、3回以上=22%)。ゲートは形式条件しか見ておらず、情景適合を判定していない。
- LLM審査段は追加しない(レイテンシ優先)。生成プロンプト内部の自己判定で代替する。
- 設計目標: **選曲の幅を狭めずにbadを減らす**。嵌めるのは情景・感情、広げるのはジャンル・国・年代。

## 変更1: 時間カードのpromptTextを情景ベースに書き換え

対象: `src/pipeline/definitions.ts` の moodCards

原則: 時間カードは「いつ・どこ(情景)」だけを主張し、音像・BPM・熱量には言及しない。それらは moodカードと heatスライダーの管轄(直交化)。

### midnight(最優先)

```
夜更けの情景を持つ曲。夜の街、ネオン、月、終電後、ひとりの部屋、バーの片隅——歌詞・タイトル・曲の世界観が「夜」を描いていることが第一条件。アレンジの静かさは条件ではない。静かでも昼や野外の情景の曲(牧歌的フォーク、青春讃歌、朝の歌)は不可。夜の情景があれば、音数が多い曲やグルーヴのある曲でもよい。
```

較正例(定義に含めるか、生成プロンプト側の【判定の較正例】に置く):
- 適合: ブルー・ライト・ヨコハマ(静かではないが夜の情景)、黄昏のビギン
- 不適合: 風をあつめて(静かだが昼の情景)、翼をください・青い山脈(青春讃歌)

### morning

```
朝の情景を持つ曲。目覚め、朝の光、始まり、出かける前の時間——世界観が「朝」または「一日の始まり」を描いていること。夜の情景・深夜の空気を持つ曲は、音が軽やかでも不可。重厚・壮大すぎる曲は朝に合わない。
```

較正例: HONNE「Crying Over You」は音がメロウでも夜の曲なので不適合。Arcade Fire「Crown of Love」は壮大すぎて不適合。

### dusk / afternoon

同じ方針で書き換え(dusk: 昼と夜の境目の情景・感傷、afternoon: 午後のゆるんだ時間の情景)。音像への言及は「補助条件にすぎない」と明記するか削る。rain / sunny_holiday は現状維持でよいが、余裕があれば同様に。

## 変更2: grit の定義境界を拡張

ユーザーのgood評価メモに基づき、gritの適合範囲を「歪み」より広く定義する:

```
ざらついた質感。歪んだギター、荒い録音、生々しいエネルギー。歪みがなくても、録音の生々しさや非ポップスな手触り(ローファイ、ガレージ的なラフさ、オルタナのカバー等)があれば適合。クリーンに整音された美しいフォークロックの名曲(例: Harvest Moon的なもの)は、名曲でも不適合。ジャンルの指定ではなく質感の軸であり、ブルースの生々しさ、初期ヒップホップの粗さ、メタルの轟音など、ジャンルを横断する。
```

例示を意図的に離れたジャンルから採ることで、例示が「柵」ではなく「軸の見本」として働くようにする(選曲の幅を守るため)。

## 変更3: カード排他の追加

definitions.ts に追加。UIの選択制御とプロンプト側検証の両方がこれを参照する(単一ソース)。

```typescript
// 排他定義
export const exclusionGroups: string[][] = [
  ["morning", "afternoon", "dusk", "midnight", "sunny_holiday"], // 時間帯はグループ内排他
];

export const exclusionPairs: [string, string][] = [
  ["doze", "dance"],
  ["doze", "grit"],
  ["doze", "uplift"],
];

export function conflictsWith(id: string, selected: string[]): string[] {
  const out = new Set<string>();
  for (const g of exclusionGroups) {
    if (g.includes(id)) selected.filter(s => s !== id && g.includes(s)).forEach(s => out.add(s));
  }
  for (const [a, b] of exclusionPairs) {
    if (a === id && selected.includes(b)) out.add(b);
    if (b === id && selected.includes(a)) out.add(a);
  }
  return [...out];
}
```

- rain は時間帯と共存しうる(雨の朝)ため排他グループに含めない。
- UI挙動: 後から選んだカードが勝ち、衝突する既選択カードは自動で外れる(押したら差し替わる)方式。
- 注: `midnight + dance` は排他にしない(ログでok 4 / bad 0。化学反応として活かす → 変更4)。

## 変更5: describeState に優先順位・正直さセクションを追加

describeState の出力末尾に固定セクションを追加:

```
【条件が両立しないとき】
- 優先順位は「ことば・地域 > 雰囲気カード > スライダー」。
- 適合曲が見つからない場合、人気度→年代の順でスライダー条件を緩めてよい。雰囲気カードは緩めない。
- 条件を緩めた場合は、選曲理由にどの条件を緩めたかを明記する。
- 選曲理由には、その曲が実際に持つ特徴だけを書く。条件に合わない曲を合うと偽って正当化しない。
```

popularity「定番」バンド(81-100)のpromptText末尾にも追記: 「ただし雰囲気カードとの適合が最優先。適合する定番曲が尽きたら、知名度を下げてでも雰囲気を守る。」

## 変更6: 生成プロンプトの骨格を変更(自己判定の内蔵)

選曲プロンプトを以下の構造にする。5候補→自己判定→1曲は同一コール内で完結させる(追加レイテンシはトークン分のみ)。

```
あなたはラジオDJとして次の曲を選ぶ。

【いまの卓の条件】
{describeState() の出力}
{comboHints 該当時は【組み合わせの解釈】}

【選曲の原則】
1. 雰囲気カードは「情景・感情」の指定であり、ジャンルの指定ではない。
   条件を満たすなら、どのジャンル・国・年代からでも選んでよい。
   定義文の例示はジャンルの範囲ではなく、軸がジャンルを横断することを示す見本である。
2. 条件が両立しないときは【条件が両立しないとき】の優先順位に従う。
3. 直近の流れと同じアーティスト・同じ質感が続いたら、条件の範囲内で意図的に離れた場所から選ぶ。

【判定の較正例】
「静かな曲」と「夜更けの曲」は別物である。風をあつめて(はっぴいえんど)は静かだが昼の情景であり midnight に不適。ブルー・ライト・ヨコハマは静かではないが夜の情景であり適合。判定は音の静かさではなく情景で行う。

【手順】
まず候補を5曲挙げる。次に各候補を、雰囲気カードの一枚ずつすべてに照らして自己判定する——その曲が実際に持つ情景・性格に基づいて。全カードを満たす曲だけを残す。「合うと言えなくもない」程度の曲は落とす。よく知らない曲は、落とさずに理由へ「推測」と明記して残してよい(無名曲は発見の源泉のため排除しない)。最も確かな1曲を選び、実際の特徴だけを理由として出力する。
```

重要な設計判断:
- 「各カードごとに照合、全カード合格」を明文化する(3枚使い時の「浮いた1枚の黙殺」対策。bad 3件の直接原因)。
- 知らない曲は「落とす」ではなく「推測と明記して通す」(選曲の幅を守る。bad 1件の残存リスクは許容)。

## 変更7: コード側フィルタ(数値ゲートの同類、レイテンシほぼゼロ)

### 7a. バージョン文字列フィルタ

Spotify解決結果のトラック名に以下を含む場合は弾き、別バージョンを再解決:

(実績: 「青い山脈 - オリジナル・カラオケ」がjudge通過してbadになった)

### 7b. 年ズレ検知

解決結果のリリース年と、選曲AIが想定した年(または既知の原曲年)が大きくずれる場合(目安: ±10年超)、別リリース(オリジナルアルバム優先)を再解決。
(実績: 異邦人が2024年盤=パイプオルガン版、ルビーの指環が2011年盤=ピアノ版を掴んでbad)

### 7c. replayのカード照合

replay(再注入)前に、対象曲の元のconditionSnapshotと現在のカード集合を照合。少なくとも地域カード(lang_ja等)が不一致なら注入しない。可能なら雰囲気カードの主軸も照合。
(実績: lang_ja指定中にreplay経由で非日本語曲が7件混入、うちbad 1件)

## 既知の別件(今回のスコープ外、余裕があれば)

- eraバンドの隙間: 2000〜2004年がどのバンドの明示レンジにも入らない(〜1975 / 1970-1999 / 2005〜 / 2018〜)。意図的でなければ 1970-2004 に修正。

- experimental × 深掘りは「実験音楽の大定番」に収束しがち。必要なら「実験音楽の教科書的名盤は避ける」を追記。

## 変更13: grit定義がtexture(エレクトロニック)を上書きしてしまう問題(2026-07-19 実運用フィードバック)

### 背景
`grit + uk`条件でtextureをエレクトロニック側(100)に設定してテスト再生したところ、選出された曲はMogwai・The Clash・Arctic Monkeys・Muse・Harry Styles等、**全曲ギター中心のロック/パンク系**で、エレクトロニックなプロダクションの曲は一つも出なかった。textureスライダーの指定が実質無視されている。

### 原因(推定)
gritの定義文(変更2)の例示が「歪んだギター」「ガレージ」「ブルース」「メタルの轟音」とロック/生楽器寄りの語彙に偏っており、エレクトロニックな"ざらつき"(インダストリアル、ビットクラッシュ、ハーシュノイズ、荒いブレイクビーツ等)への言及が一切ない。モデルが"ざらつき"を反射的にギターと結びつけ、texture=エレクトロニックの指定より例示の重力を優先してしまっている。

grit(質感の粗さ)とtexture(生楽器⇄電子)は本来直交する軸であり、矛盾しない(例: インダストリアル、EBM、ハーシュノイズ系テクノは"電子的でざらついた"音楽として実在する)。設計意図としては両立可能なはずが、定義文の偏りで片方が事実上死んでいる状態。

### 対策(実施済みだが効果なしと判明): gritの定義文にエレクトロニックな例を追加

```
grit(改訂版、2026-07-19実装・デプロイ済み): "ざらついた質感。歪んだギター、荒い録音、生々しいエネルギー。
歪みがなくても、録音の生々しさや非ポップスな手触り(ローファイ、ガレージ的な
ラフさ、オルタナのカバー等)があれば適合。クリーンに整音された美しい
フォークロックの名曲(例: Harvest Moon的なもの)は、名曲でも不適合。
ジャンルの指定ではなく質感の軸であり、ブルースの生々しさ、初期ヒップホップの
粗さ、メタルの轟音、インダストリアルの歪んだシンセ、ビットクラッシュされた
電子ドラム、ハーシュノイズ系テクノの荒さなど、生楽器・電子楽器を問わず
ジャンルを横断する。textureがエレクトロニック側に振られている場合、
ギターではなく電子的な歪み・粗さの側面でgritを満たす曲を選ぶこと。"
```

### 実運用テスト結果(2026-07-19): 効果なし
本番デプロイ済み(git merge-base --is-ancestorで確認、Vercelデプロイも変更13・14を含むことを確認済み)の状態で`grit + uk + texture=100`を再テストしたが、**Arctic Monkeys、Blur、The Kooks、The Smiths、The Ting Tings等、依然として全曲ギター中心のロック**。policySnapshotには「**エレクトロニックな要素は避け、生楽器主体の粗さを追求します**」と、texture=100の指定と真逆の方針が明記された。

**結論: プロンプト文言への追記(電子的な例を足す)だけでは、gritの"ギター"という連想を覆せない。** これは変更12(国籍)で「文言追加→効果薄い→コード側ハード検証」という流れを辿ったのと同じ壁。「ざらつき」という言葉自体が、モデルの学習データの中でロック/ギタージャンルの語りと圧倒的に強く結びついており、定義文をどう書いても完全には覆せない可能性が高い。

### 次の一手: textureが極端な値のとき、gritの定義自体をコード側で動的に差し替える(次回実装)

文言強化(prompt内に追記)では覆せなかったので、**gritのpromptText自体を、texture値に応じてコード側で条件分岐させる**設計に切り替える。単一の固定文字列ではなく、生成関数にする。

```typescript
// definitions.ts
// gritはtexture値に応じて説明文の重心を動的に変える。
// 通常の(電子例を後方に添えるだけの)固定文言では、ギターへの言及が
// 先に来ることで連想が強く残ってしまうため(2026-07-19実測で確認済み)、
// texture極端値のときは電子的な説明を「主」、ギター的な説明を「例外」として
// 前後を入れ替える。
function getGritPromptText(textureValue: number): string {
  if (textureValue >= 80) {
    // エレクトロニック優勢: 電子的な粗さを主役にし、ギターは終盤に触れるのみ
    return "ざらついた質感。インダストリアルの歪んだシンセ、ビットクラッシュ" +
      "された電子ドラム、ハーシュノイズ系テクノの荒さ、荒いブレイクビーツなど、" +
      "電子的な歪み・粗さを最優先で探すこと。歪んだギターのロック/パンクは" +
      "この条件ではむしろ避ける(textureがエレクトロニック指定のため)。" +
      "クリーンに整音されたポップなシンセ(テクノポップの名曲等)は、" +
      "名曲でも不適合。ジャンルの指定ではなく質感の軸である。";
  }
  if (textureValue <= 20) {
    // アコースティック優勢: 従来通りギター/生楽器中心の説明(変更2のまま)
    return GRIT_ACOUSTIC_PROMPT_TEXT; // 既存の変更2定義文をそのまま使用
  }
  // 中間値: 両論併記(現行の変更13定義文をそのまま使用)
  return GRIT_NEUTRAL_PROMPT_TEXT;
}
```

`describeState`および`generateCandidates`の両方で、gritカードのpromptTextを参照する箇所を、この関数の呼び出しに置き換える(単一ソース設計を維持: 呼び出し元を増やさず、関数を1箇所に集約する)。

ポイントは、**「電子的な例を後から付け足す」のではなく、「電子優勢のときは電子的な説明を主役にして、ギターの話を意図的に後回しにする、あるいは明示的に除外する」**という発想の転換。今回の実測(文言追加だけでは効果なし)を踏まえ、"言葉の位置と量"そのものを動的に操作する。

### 背景(2026-07-19 追加調査で判明)
9-11(文言追加)実装後も`uk`単体テストで再発(The Strokes=アメリカが選出、candidateWhyに国籍言及なし)。ローカルのコード調査(core.ts, route.ts)で原因が判明:

- `generateCandidates`は**1回のLLM呼び出し**で「候補生成→自己判定→passed/rejected」を全て行う設計。過去に別役の審査パス(LLM二段構成)を試したが、審査を厳しくしすぎて「知ってる曲=定番でNG、知らない曲=不明でNG」のデッドロックを起こした教訓と、却下回数と人間評価が無相関だった実測から、**意図的に別役の審査パスを廃止した経緯がある**(core.tsコメントに明記)。
- つまり「却下と選出が別々の推論」なのではなく、**同じ1回の推論の中で、モデルの自己判定がその回だけ機能しなかった**というのが実体。judgeRejectionsで他の曲を正しく却下していても、最終選出の一貫性は保証されない。
- fill route側(route.ts)には多層フォールバック(L1〜L5)があるが、**passed後のコード側チェックは重複排除・年代ゲート・タイトル解決のみで、国籍/地域の整合性は一度も再検証されない**。resolveTrackはSpotify上でアーティスト名が一致しさえすれば無条件で解決に成功する。
- **既知の別の穴**: L5(種曲フォールバック、fallback.ts)は固定リストからの選出で、**地域カードとの照合が一切ない**。今回のログはsource:"ai"だったためL5は無関係だが、地域偽装の別ルートとして存在する(要フォローアップ、本変更のスコープ外として別途記録)。

結論: プロンプト文言の強化だけでは、モデルの自己判定が外れた場合に検出・修正する手段がコード側に皆無。**コード側にハード検証を追加する**方向に切り替える。

### 対策: candidateWhy/rejectedのテキストから国籍宣言を抽出し、コード側でハード判定する
新しい外部API呼び出しは追加しない(レイテンシほぼゼロ)。モデルが返すテキストをコードで検証する二段構え:

1. **プロンプト側(9-11の延長)**: 地域カード指定時、各候補のcandidateWhyに**必ず**「国籍/活動拠点: ○○」という定型文言を含めることを、より強い表現で義務付ける(「省略不可」「この記載がない場合は不合格として扱われる」等、モデルへの強制力を上げる)。
2. **コード側(新規)**: `generateCandidates`が返したcandidateWhyの文字列から「国籍/活動拠点: ○○」を正規表現等で抽出し、指定されている地域カードの範囲と一致するか機械的にチェックする。一致しない場合、または該当の記載自体が存在しない場合は、その候補を**コード側で強制却下**し、次点候補を採用するか再生成を要求する。

```
【手順】(9-11からの改訂・強い表現に変更)
まず候補を5曲挙げる。次に各候補を、選ばれている性格カード・地域
カードのすべてに照らして自己判定する。地域・言語カードが指定されて
いる場合、各候補について必ず「国籍/活動拠点: ○○」という記載を
含めること(省略不可)。指定地域と一致しない場合、理由を問わず
(知名度・影響力・音楽的つながりの深さを含め)不合格とする。
全カードを満たす曲だけを残し、最も確かな1曲を選ぶ。
```

コード側の検証イメージ(route.tsまたはcore.ts、resolveTrack前後に追加):
```typescript
// 地域カードが指定されている場合、candidateWhyから国籍宣言を抽出しハード検証
function verifyRegionDeclaration(candidateWhy: string, regionCardId: string): boolean {
  const match = candidateWhy.match(/国籍\/活動拠点[:：]\s*(.+?)[。.\n]/);
  if (!match) return false; // 記載なし→不合格
  const declared = match[1];
  return regionMatches(declared, regionCardId); // 地域カードごとの許容国名リストと照合
}
```
`regionMatches`は各地域カード(uk, usa, lang_ja等)に対応する国名・地域名の許容リストを持つ簡易な文字列照合で十分(厳密な国籍データベースは不要)。

### 既知の限界
モデルが国籍宣言自体を書かなかった場合(記載なし)は「不合格」として扱えるため、今回のThe Strokesのケース(国籍言及ゼロ)はこの仕組みで検出可能になる。ただし国籍を誤って(虚偽で)申告した場合は、依然として検出できない——この対策は「書かせて、書いたものを機械的にチェックする」ものであり、モデルの申告内容そのものの真偽までは保証しない。完全な確実性が必要ならMusicBrainz等の外部データソースとの照合(処理コスト増)を別途検討する。

### 9-1. 性格カードの選択上限を2枚に固定

時間・シーン・雰囲気カードは合計で**最大2枚まで**。地域・言語カードはこの上限にカウントしない(9-3参照)。

理由: 3枚以上になると条件が過飽和し、モデルは全条件を薄く満たす無難な曲(ok評価)に収束する。2枚までなら「浮いた1枚が黙って無視される」問題(変更6の主因)も構造的に起きなくなる。

**重要(2026-07-14 実装バグ修正の訂正、さらに2026-07-15 再訂正)**: 「性格カード合計2枚まで」は、時間・シーン・気分の3グループを**横断して**合計2枚という意味であり、グループごとに1枚という意味ではない。初回実装では誤って「グループごとに1枚」になっていたため、気分・質感グループから2枚選べない不具合が発生した。正しい仕様:

- 「時間・天気」グループ(morning/midnight/rain/summer)は**グループ内排他**(実質1枚まで。同時に朝と夜更けは選べない、という情景上の制約のため)
- 「シーン」(drive/focus)も**グループ内排他**(実質1枚まで。ドライブ=前に進む推進力と、focus=一定の質感が続く集中作業は、情景として同時に成立しにくいため)
- 「気分・質感」(dance/doze/grit/experimental/bittersweet/uplift。nostalgiaは2026-07-15付で削除)には**グループ内排他なし**。同じグループから2枚選んでも構わない(例: dance+uplift、grit+bittersweetは可)。これは初回実装のままで正しかった(一時「グループごとに1枚では」と疑ったが撤回)
- 上記全グループを横断して、選択できる性格カードの**合計は2枚まで**
- 個別の排他ペア(doze×dance、doze×grit、doze×uplift)は上記とは別に維持

### 9-2. 性格カードの棚卸し(20枚→13枚)

較正・議論の結果、以下の通り整理。**削除したカードの情景・感情は、原則として無選択(自由に選んでよい)かheat/textureスライダー、または近縁カードに吸収される設計とする。**

**時間・天気(6→4枚)**
- 残す: `morning`(朝)、`midnight`(夜更け。変更1で情景ベースに書き換え済み)、`rain`(雨、変更なし)
- 削除: `afternoon` — 無選択時のデフォルト("自由に選んでよい")とほぼ等価なため独立カードの必要性が薄い
- 削除: `dusk` — 「感傷的で美しい」という情景がbittersweetと重複。bittersweetの定義に統合(下記9-2追記参照)
- 置換: `sunny_holiday` → `summer`(夏)に差し替え。「晴れた休日」は愛着が薄く、"夏"のほうが強い季節性・文化的磁場を持つため。日本の夏(蝉・花火・シティポップ)に限定せず、アメリカの夏(Summer in the City、Beach Boys的な西海岸/都会の熱気)も含む国・ジャンル非依存の定義にする:
  ```
  summer: "夏の情景。強い日差し、暑さ、開放感——都会のアスファルトの熱気でも、波打ち際の解放感でも、蝉と花火の湿った夏でも、季節としての夏を描いていればよい。国やジャンルを問わず、"夏"という季節感を持つ曲。"
  ```

**シーン(4→2枚)**
- 残す: `drive`(ドライブ、変更なし)、`focus`(作業・集中、変更なし)
- 削除: `kitchen`、`gathering` — 機能面では他と代替不可能ではないが(heat×danceやheat×upliftでかなり再現できる)、判断の決め手はUXだった: driveが"動"、focusが"静"で両極を代表しており、kitchen/gatheringはその中間で存在感が薄い。「一人か人がいるか」という文脈の違いは実在するが、選曲への影響はheatスライダーがかなりの部分を代替する。

**気分・質感(10→7枚)**
- 残す: `dance`、`doze`、`grit`(変更2で境界拡張済み)、`experimental`(いずれも変更なし、他と重複しない固有の軸を持つと判断)
- 残す: `uplift` — 一度「heat×danceで代替可能では」と疑ったが、upliftが指定するのは"曲の中の起伏の構造"(ビルドアップして解放される)であり、heat(感情の温度)ともdance(リズム)とも独立した軸と判明。EDM的な高揚感はheat×danceだけでは出ない(heatのホット端は「熱唱・汗」という生々しい肉体性の軸で、EDMのひんやりした多幸感とは逆方向)。撤回して残留。
- 残す: `bittersweet` — duskの「昼と夜の境目の感傷」を吸収する形で定義を拡張:
  ```
  bittersweet: "甘さと痛みが同居する感情。マイナーとメジャーの揺らぎ、郷愁を誘うメロディ。夕暮れのような、昼と夜の境目の感傷も含む。"
  ```
- 削除(2026-07-15追記): `nostalgia` — bittersweetとの役割重複が実運用で判明(「郷愁を誘うメロディ」と「世代の記憶に触れる」がほぼ同じ意味になっていた)。年代スライダー(era)と連動させる案(「eraで指定した年代の当時の音らしさ」)は、era設定によってはeraの言い換えになってしまい自己矛盾。「実際のリリース年に関わらず意図的に懐古を演じている曲(シティポップ・リバイバル等)」という独立軸としての再定義案も検討したが、日常使いにはニッチすぎると判断し削除で確定。感情としての郷愁はbittersweetでカバーする。
- 削除: `uplift`系と誤読しないよう明記——削除したのは `romance`、`nature`、`city` の3枚:
  - `romance`: 「甘く親密」はbittersweet寄りの感情+heat低〜中で代替可能。ジャンル指定(ソウル、ボサノバ)に寄りすぎていた定義の粗さも要因。
  - `nature`: 「アコースティックな手触り」はtextureスライダー(9-4で新設)のアコースティック端とほぼ等価。
  - `city`: 「シティポップ、ネオソウル」はジャンル指定に近く、むしろ地域・都市文化の指定に近い。heat低め×textureエレクトロニック寄りでかなり再現可能。ただし「Take Fiveのようなジャズが出したいときの受け皿」を失う副作用があり、これがtextureスライダー新設の直接のきっかけになった。

**保留(次の季節が来たら追加)**
- `christmas`(クリスマス): 夏とは性質が異なる「特定の日付に紐づく祝祭」カード。常設ではなく季節限定で追加する運用とし、今回のスコープには含めない。

## 変更15: verifyRegionDeclarationの逆接語バグ修正(2026-07-19 実運用フィードバック)

### 背景
変更13続き(grit動的差し替え)デプロイ後の`uk + grit + texture=100`実運用テストで、Travis Scott(アメリカ)が選出された:

```
candidateWhy: "国籍/活動拠点: アメリカだが、イギリスで数回パフォーマンスを
行っており、UKガラージに影響を与えたアーティストとしての評価がある。
電子的な歪みと多層的なビートを特徴とする。"
```

モデル自身は**正しく「アメリカ」と断言しており、虚偽申告ではない**。問題はコード側にあった: `regionMatches`が抽出した宣言文字列全体に対して単純な部分文字列一致(`.includes()`)を行っていたため、「アメリカだが、イギリスで…影響を与えた」という、変更9-11で禁止したはずの"影響力"型の正当化が後に続くと、その中の「イギリス」「UK」にヒットして誤って合格判定になっていた。

同じテストで、Bring Me The Horizon(実際はイギリスのバンド)自身の他曲やThe Prodigy(イギリス)が、逆に「イギリスのアーティストであるため不合格」という趣旨の理由でjudgeRejectionsに却下されている観察もあった。これはモデル自身の自己判定ロジックの混乱であり、今回は実害(bad選出)には至っていないため対応は見送るが、記録として残す。

### 対策: 宣言文を逆接語で分割し、最初に断言した部分だけを判定に使う

```typescript
const HEDGE_SPLIT_RE = /だが|けど|しかし|ただし|もっとも/;

export function verifyRegionDeclaration(candidateText: string, regionCardId: string): boolean {
  const match = String(candidateText ?? "").match(REGION_DECLARATION_RE);
  if (!match) return false;
  const primary = match[1].split(HEDGE_SPLIT_RE)[0];
  return regionMatches(primary, regionCardId);
}
```

逆接語のリストは今回判明した「だが/けど/しかし/ただし/もっとも」のみとする。「とはいえ」「ではあるものの」等、別の逆接表現で同種の問題が再発する可能性はあるが、網羅的なリスト化はせず、実際に見つかった時点でリストに追加する運用とする(既存のKNOWN_ARTIST_NATIONALITY_OVERRIDESと同じ「都度追加」方針)。

### 検証結果(2026-07-19、APIを叩かない検証スクリプトで確認)
- Travis Scottの実例: `false`(却下)に修正されることを確認
- 既存の回帰ケース(The Strokesの記載なし、正しいUK宣言、The Killersの虚偽申告→補正リスト側で別途弾く対象、usa宣言、world_tripの除外)は全て従来どおりの結果を維持
- 逆接語「けど」でも同様に弾けることを確認
- 正しい宣言の直後の句点で文が終わる場合(逆接語が別文にある場合)は影響を受けないことを確認

## 変更17: VERSION_MARKERSに所有格+Mix、別バージョン系複合語を追加(2026-07-19 実運用フィードバック)

### 背景
同じ実運用テストで、The Stone Rosesの候補が実際には`"Fools Gold - Grooverider's Mix"`(リミックスアルバム『The Remixes』収録)として解決された。`resolveMeta.titleExact: false`が示す通り、意図した原曲ではない。既存の変更7a(VERSION_MARKERS)は`remix`/`リミックス`は捕捉するが、`"〜's Mix"`という**remixという単語を使わない別バージョン表記**は捕捉できていなかった。

### 検討: 裸の"mix"を追加しない理由
ユーザーからの指摘で、単純に`mix`をキーワードとして追加すると誤爆することが判明した:
- `"Mixed Emotions"`(実在する曲名)を誤って除外してしまう
- `"Original Mix"`(ダンス/エレクトロニック曲で原曲を示す正規のタグ)を誤って除外してしまう——`grit + texture=100`はエレクトロニック系の曲を集めやすい条件のため、この誤爆は実害が大きい

### 対策
実際に問題になったパターンだけを狙い撃ちする:

```typescript
const VERSION_MARKERS =
  /remix|live|edit|acoustic|instrumental|karaoke|demo|sped up|slowed|cover|version|take\s*\d+|outtake|alternate|anthology|rehearsal|['’]s\s+mix|radio\s+mix|club\s+mix|extended\s+mix|dub\s+mix|カラオケ|インストゥルメンタル|インスト|ライブ|ライヴ|リミックス/i;
```

- `['’]s\s+mix`: 所有格(リミキサー名)+ Mix。直/カーリー両方のアポストロフィに対応。今回の実例`"Grooverider's Mix"`を捕捉する。
- `radio mix` / `club mix` / `extended mix` / `dub mix`: 業界で確立された別バージョン系の複合語。裸の`mix`のような一般名詞ではなく、実在曲名との衝突リスクが低い具体的な複合語のみ追加した。

### 検証結果(2026-07-19、APIを叩かない検証スクリプトで確認)
11ケースの回帰テストで全て合格:
- 除外すべき: `"Grooverider's Mix"`(直/カーリー両方)、`"(Radio Mix)"`、`"(Club Mix)"`、`"(Extended Mix)"`、`"(Dub Mix)"`
- 誤爆してはいけない: `"Mixed Emotions"`、`"Original Mix"`、`"Six Mixtapes"`、`"Admix"`、通常のオリジナルタイトル(`"Fools Gold"`)

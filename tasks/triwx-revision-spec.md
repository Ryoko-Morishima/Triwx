# TRIWX 選曲品質改修 指示書

作成日: 2026-07-13
根拠: 再生ログ150セグメント(11セッション、人間評価41件: good 26 / ok 28 / bad 15)の分析

## 背景(要約)

- bad 15件の主因は3系統: (a) 時間カードの定義がアレンジベースで情景とずれる(midnight事故9件)、(b) 3枚使い時に浮いた1枚が黙って無視される(grit不足3件)、(c) パイプライン起因(replayのカード無照合、Spotify解決のバージョン事故)。
- 数値ゲート(judge)の却下回数と人間評価は無相関(bad率: 却下0回=25%、1-2回=21%、3回以上=22%)。ゲートは形式条件しか見ておらず、情景適合を判定していない。
- LLM審査段は追加しない(レイテンシ優先)。生成プロンプト内部の自己判定で代替する。
- 設計目標: **選曲の幅を狭めずにbadを減らす**。嵌めるのは情景・感情、広げるのはジャンル・国・年代。

---

## 変更1: 時間カードのpromptTextを情景ベースに書き換え

対象: `src/pipeline/definitions.ts` の moodCards

原則: 時間カードは「いつ・どこ(情景)」だけを主張し、音像・BPM・熱量には言及しない。それらは moodカードと heatスライダーの管轄(直交化)。

### midnight(最優先)

```
夜更けの情景を持つ曲。夜の街、ネオン、月、終電後、ひとりの部屋、バーの片隅——歌詞・タイトル・曲の世界観が「夜」を描いていることが第一条件。アレンジの静かさは条件ではない。静かでも昼や野外の情景の曲(牧歌的フォーク、青春讃歌、朝の歌)は不可。夜の情景があれば、音数が多い曲やグルーヴのある曲でもよい。
```

較正例(定義に含めるか、生成プロンプト側の【判定の較正例】に置く):
- 適合: ブルー・ライト・ヨコハマ(静かではないが夜の情景)、黄昏のビギン
- 不適合: 風をあつめて(静かだが昼の情景)、翼をください・青い山脈(青春讃歌)

### morning

```
朝の情景を持つ曲。目覚め、朝の光、始まり、出かける前の時間——世界観が「朝」または「一日の始まり」を描いていること。夜の情景・深夜の空気を持つ曲は、音が軽やかでも不可。重厚・壮大すぎる曲は朝に合わない。
```

較正例: HONNE「Crying Over You」は音がメロウでも夜の曲なので不適合。Arcade Fire「Crown of Love」は壮大すぎて不適合。

### dusk / afternoon

同じ方針で書き換え(dusk: 昼と夜の境目の情景・感傷、afternoon: 午後のゆるんだ時間の情景)。音像への言及は「補助条件にすぎない」と明記するか削る。rain / sunny_holiday は現状維持でよいが、余裕があれば同様に。

## 変更2: grit の定義境界を拡張

ユーザーのgood評価メモに基づき、gritの適合範囲を「歪み」より広く定義する:

```
ざらついた質感。歪んだギター、荒い録音、生々しいエネルギー。歪みがなくても、録音の生々しさや非ポップスな手触り(ローファイ、ガレージ的なラフさ、オルタナのカバー等)があれば適合。クリーンに整音された美しいフォークロックの名曲(例: Harvest Moon的なもの)は、名曲でも不適合。ジャンルの指定ではなく質感の軸であり、ブルースの生々しさ、初期ヒップホップの粗さ、メタルの轟音など、ジャンルを横断する。
```

例示を意図的に離れたジャンルから採ることで、例示が「柵」ではなく「軸の見本」として働くようにする(選曲の幅を守るため)。

## 変更3: カード排他の追加

definitions.ts に追加。UIの選択制御とプロンプト側検証の両方がこれを参照する(単一ソース)。

```typescript
// 排他定義
export const exclusionGroups: string[][] = [
  ["morning", "afternoon", "dusk", "midnight", "sunny_holiday"], // 時間帯はグループ内排他
];

export const exclusionPairs: [string, string][] = [
  ["doze", "dance"],
  ["doze", "grit"],
  ["doze", "uplift"],
];

export function conflictsWith(id: string, selected: string[]): string[] {
  const out = new Set<string>();
  for (const g of exclusionGroups) {
    if (g.includes(id)) selected.filter(s => s !== id && g.includes(s)).forEach(s => out.add(s));
  }
  for (const [a, b] of exclusionPairs) {
    if (a === id && selected.includes(b)) out.add(b);
    if (b === id && selected.includes(a)) out.add(a);
  }
  return [...out];
}
```

- rain は時間帯と共存しうる(雨の朝)ため排他グループに含めない。
- UI挙動: 後から選んだカードが勝ち、衝突する既選択カードは自動で外れる(押したら差し替わる)方式。
- 注: `midnight + dance` は排他にしない(ログでok 4 / bad 0。化学反応として活かす → 変更4)。

## 変更4: comboHints(化学反応する組み合わせの追加解釈)

網羅しない。LLMの自然解釈が外しそうな組だけ少数登録し、テストで「ヒント不要」とわかったら削る運用。

```typescript
export const comboHints: { cards: string[]; promptText: string }[] = [
  {
    cards: ["midnight", "dance"],
    promptText: "深夜のクラブ・フロアの時間帯。ディープハウス、テクノ、夜が深まるほど効くグルーヴ。昼のパーティーチューンではなく、暗さと快楽が同居する曲。",
  },
  {
    cards: ["bittersweet", "dance"],
    promptText: "泣きながら踊る音楽。切なさと踊れるビートの同居(Robyn的な様式、本来のディスコが持っていた感情)。切ないミッドテンポに丸めない。",
  },
  {
    cards: ["rain", "dance"],
    promptText: "雨の都市のダンスミュージック。UKガラージ、ダブステップ、しっとりした質感とビートの共存。",
  },
  {
    cards: ["morning", "bittersweet"],
    promptText: "徹夜明けの朝、昨夜を引きずった朝。清潔な朝ではなく、カムダウンの時間。",
  },
];
```

describeState でカード集合が cards をすべて含む場合、【組み合わせの解釈】セクションとして出力に追加する。

## 変更5: describeState に優先順位・正直さセクションを追加

describeState の出力末尾に固定セクションを追加:

```
【条件が両立しないとき】
- 優先順位は「ことば・地域 > 雰囲気カード > スライダー」。
- 適合曲が見つからない場合、人気度→年代の順でスライダー条件を緩めてよい。雰囲気カードは緩めない。
- 条件を緩めた場合は、選曲理由にどの条件を緩めたかを明記する。
- 選曲理由には、その曲が実際に持つ特徴だけを書く。条件に合わない曲を合うと偽って正当化しない。
```

popularity「定番」バンド(81-100)のpromptText末尾にも追記: 「ただし雰囲気カードとの適合が最優先。適合する定番曲が尽きたら、知名度を下げてでも雰囲気を守る。」

## 変更6: 生成プロンプトの骨格を変更(自己判定の内蔵)

選曲プロンプトを以下の構造にする。5候補→自己判定→1曲は同一コール内で完結させる(追加レイテンシはトークン分のみ)。

```
あなたはラジオDJとして次の曲を選ぶ。

【いまの卓の条件】
{describeState() の出力}
{comboHints 該当時は【組み合わせの解釈】}

【選曲の原則】
1. 雰囲気カードは「情景・感情」の指定であり、ジャンルの指定ではない。
   条件を満たすなら、どのジャンル・国・年代からでも選んでよい。
   定義文の例示はジャンルの範囲ではなく、軸がジャンルを横断することを示す見本である。
2. 条件が両立しないときは【条件が両立しないとき】の優先順位に従う。
3. 直近の流れと同じアーティスト・同じ質感が続いたら、条件の範囲内で意図的に離れた場所から選ぶ。

【判定の較正例】
「静かな曲」と「夜更けの曲」は別物である。風をあつめて(はっぴいえんど)は静かだが昼の情景であり midnight に不適。ブルー・ライト・ヨコハマは静かではないが夜の情景であり適合。判定は音の静かさではなく情景で行う。

【手順】
まず候補を5曲挙げる。次に各候補を、雰囲気カードの一枚ずつすべてに照らして自己判定する——その曲が実際に持つ情景・性格に基づいて。全カードを満たす曲だけを残す。「合うと言えなくもない」程度の曲は落とす。よく知らない曲は、落とさずに理由へ「推測」と明記して残してよい(無名曲は発見の源泉のため排除しない)。最も確かな1曲を選び、実際の特徴だけを理由として出力する。
```

重要な設計判断:
- 「各カードごとに照合、全カード合格」を明文化する(3枚使い時の「浮いた1枚の黙殺」対策。bad 3件の直接原因)。
- 知らない曲は「落とす」ではなく「推測と明記して通す」(選曲の幅を守る。bad 1件の残存リスクは許容)。

## 変更7: コード側フィルタ(数値ゲートの同類、レイテンシほぼゼロ)

### 7a. バージョン文字列フィルタ

Spotify解決結果のトラック名に以下を含む場合は弾き、別バージョンを再解決:
`カラオケ / Karaoke / Instrumental / インストゥルメンタル / Live / ライブ / ライヴ / Remix / リミックス / cover(自アーティスト以外) / Acoustic Version` 等。
(実績: 「青い山脈 - オリジナル・カラオケ」がjudge通過してbadになった)

### 7b. 年ズレ検知

解決結果のリリース年と、選曲AIが想定した年(または既知の原曲年)が大きくずれる場合(目安: ±10年超)、別リリース(オリジナルアルバム優先)を再解決。
(実績: 異邦人が2024年盤=パイプオルガン版、ルビーの指環が2011年盤=ピアノ版を掴んでbad)

### 7c. replayのカード照合

replay(再注入)前に、対象曲の元のconditionSnapshotと現在のカード集合を照合。少なくとも地域カード(lang_ja等)が不一致なら注入しない。可能なら雰囲気カードの主軸も照合。
(実績: lang_ja指定中にreplay経由で非日本語曲が7件混入、うちbad 1件)

## 変更8: ログの拡充(検証インフラ)

再テストの効果測定のために、セグメントログへ以下を追加:

- `model`: 選曲に使ったモデル名(現状未記録。プロンプト効果とモデル効果の切り分けに必須)
- `promptVersion`: プロンプトのバージョンタグ
- `judgeRejections`: 却下の回数だけでなく `{track, reason}` の配列(却下理由の分析用)
- 生成コール内の自己判定で落ちた候補と理由(可能なら。5候補→1曲の脱落ログは定義改良の材料になる)

## 既知の別件(今回のスコープ外、余裕があれば)

- eraバンドの隙間: 2000〜2004年がどのバンドの明示レンジにも入らない(〜1975 / 1970-1999 / 2005〜 / 2018〜)。意図的でなければ 1970-2004 に修正。
- nostalgia の基準世代が未定義(「誰の世代か」)。テストで違和感が出たら基準年代を明示するか era と連動させる。
- experimental × 深掘りは「実験音楽の大定番」に収束しがち。必要なら「実験音楽の教科書的名盤は避ける」を追記。

## 再テスト計画

1. 改修後、**同一条件での再テスト**を先に行う: `midnight + lang_ja`(era=0, popularity=100)と `dusk + nostalgia + grit`(era=52)。bad率の差分を見る。
2. 次に同方向3枚(morning+grit+uplift型)をもう1〜2組。
3. 判断に自信のない組み合わせを少数——採点データとしては弱いが、「採点せずに聴いて楽しいか」を測る(こちらが本来の目的)。

## 机上検証の結果(参考)

bad 15件のうち、本指示書の対策での防止見込み:
- 情景定義+較正例: 5件(RADWIMPS、BEGIN、花の首飾り、風をあつめて、翼をください)
- 各カード個別照合: 4件(Crown of Love、Summer Breeze、青い影、Harvest Moon)
- morning情景化: 1件(HONNE)
- コード側: 4件(カラオケ=文字列、Silk City=replay照合、異邦人・ルビーの指環=年ズレ)
- 残存リスク許容: 1件(風の行方=無名曲の知識不足。幅を守るため「推測明記で通す」を選択)

good 26件への誤爆チェック済み。夜に駆ける・地上の星・ブルー・ライト・ヨコハマは新定義(情景ベース)で生存。grit境界例(Beirut、The Watson Twins)は変更2の境界拡張で生存。

# HARDWARE.md — TRIWX ハードウェア化 検討資料

> 将来構想。TRIWXを物理のラジオ端末として使えるようにするための設計検討とロードマップ。
> コードはまだ書かない。着手時は本書を読んでから tasks/ に起票すること。
> 最終更新: 2026-07-05

## ビジョン

部屋の棚に木の筐体のラジオが置いてあり、電源を入れると音楽が流れはじめる。
ノブを回すと選曲の空気が変わり、ランプが曲間に灯り、針が曲の進行を指す。
画面を見なくても、手と耳だけで完結するTRIWX。

## 絶対原則（ソフトと同じ、むしろ物理ではより厳しく）

**針・ランプ・ノブ・メーター・音は、飾りではなく本物の状態や操作に対応すること。**
Webの飾りダイアルは1クリックで直せたが、物理の嘘は木工からやり直しになる。
部品を1つ足すたびに「これは何の実状態を表示するのか / 何の実操作を送るのか」に答えられなければ載せない。

各部品と「本物」の対応表（設計の出発点）:

| 物理部品 | 対応する本物 | 備考 |
|---|---|---|
| 電源スイッチ | 放送の開始/停止 | 開局サウンドロゴもここから |
| ロータリーノブ×3 | 年代感・温度感・人気度フェーダーの実値 | 回転→state変更→warm発火。既存のstateVersion機構がそのまま受ける |
| プリセットボタン数個 | 保存済みカードセットの呼び出し | 物理で27枚のカードは非現実的。よく使う組合せ（例:「ざらつき+切なさ+夕暮れ」）をプリセット化する。**プリセット保存機能はソフト側の前提作業** |
| スキップボタン | engine.skip() | |
| ◎ / ✕ ボタン | 評価（カード適合） | 物理ボタンの評価は体験として最高に「ラジオ局」 |
| ランプ（LED） | 停波/音楽/マイクの3状態（現行ランプと同一の意味論） | |
| アナログメーター（可動針） | 曲の経過時間（現行の文字盤と同じ意味） | サーボまたは電流計型ムーブメント |
| VUメーター | **実際の音声出力レベル** | ブラウザでは不可能だった「本物のVU」が、実機なら音声信号のタップで実現できる。ハードウェア化の音響的ハイライト |
| スピーカー | 実再生の出力 | |

## アーキテクチャ段階案

### H0: キオスク化（ハンダ付けゼロ・最小投資で「置き型ラジオ」）
- Raspberry Pi（4以降推奨）+ 小型ディスプレイ + USB DAC/アンプ + スピーカー
- 既存のWebアプリをそのまま Chromium キオスクモードで全画面表示。コード変更ほぼ不要
- **要検証（着手時の最初の確認事項）**: Raspberry Pi の Chromium で Spotify Web Playback SDK が動くか。SDKはDRM（Widevine）必須で、Pi向けWidevineの導入と安定性は環境依存。**ここが通らなければH0の設計を変える**（下記代替案）
- 代替案: librespot（PiをSpotify Connectデバイス化するOSS）+ 再生指示をWeb SDKからConnect API（play on device_id）に切替。技術的には筋が良いが、librespotは非公式実装でありSpotifyの規約上グレー。採用judgment は人間（局長）の判断事項
- 自動起動・自動ログイン（トークン永続）の整備が必要

### H1: 物理操作（ノブ・ボタン）
- GPIO + ロータリーエンコーダ + ボタン → Pi上の小さな常駐デーモン（Node/Python）が読み取り
- デーモン→アプリの伝達路: **「コントロールAPI」の新設が前提作業**（例: ローカルWebSocketか `/api/control` で fader値/skip/評価/プリセットを受ける口）。現行アプリはUI操作のみを想定しているため、ここはソフト側の設計が先
- ノブは絶対値ポテンショメータか相対エンコーダかで設計が変わる（電源投入時のズレ問題）。推奨は相対エンコーダ+起動時は現在値から

### H2: 物理表示（ランプ・メーター）
- LED: エンジンの状態（playing/narrating/starved）をコントロールAPI経由でデーモンへプッシュ → GPIO点灯。現行のmicFlash相当も
- 針メーター: 曲の経過%をサーボへ。更新は1〜2秒間隔で十分
- VU: DAC出力の音声信号をエンベロープ検出（回路 or ソフトでRMS）→ メーター駆動。**実信号なので原則に完全適合**

### H3: 筐体
- 木の筐体、クリームの文字盤パネル、真鍮調ノブ——WebのUIが物理の設計図になる
- ここは完全に局長の工作・趣味の領分。AIは寸法図・部品表・配線図の作成で支援

## 役割分担

- **人間が判断・実施**: 予算と部品購入、librespot採否（規約判断）、組み立て・木工・配線、音の最終評価
- **AIが実装**: プリセット保存機能、コントロールAPI、GPIOデーモン、キオスク起動スクリプト、状態プッシュ（WebSocket）、寸法図・配線図・手順書
- **共同**: H0のWidevine検証（AIが手順を書き、人間がPi上で実行）

## リスクと未知数（着手前に潰す順）

1. Pi + Chromium + Widevine で Web Playback SDK が安定動作するか（H0の生死を決める）
2. 常時稼働の熱・電源・ネットワーク断（starved状態の物理表示——ランプ点滅等——を最初から設計に入れる）
3. トークンの自動リフレッシュが無人運転で回り続けるか（現行はブラウザのCookie前提。デバイス常駐なら保存方式の見直しが要るかもしれない）
4. Spotify規約（1アカウント1ストリーム。スマホで聴きながら実機でも、は不可）
